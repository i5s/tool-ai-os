## 1. Critical Findings

### C1 тАФ Unawaited Async Provider Calls Crash at Runtime

**File:** `toll/application/notebook_service.py:143,250`  
**Type:** Bug

`NotebookPort` declares methods as `async def`, but `NotebookService` calls them from synchronous `def` methods **without** `await`:

```python
resp = self.notebook_provider.upload_source(...)   # returns coroutine, not NotebookResponse
if not resp.success:                                # AttributeError: 'coroutine' has no 'success'
```

Same pattern at line 250 for `self.notebook_provider.query(...)`.

**Trigger conditions:** `notebooklm_enabled=True` + `notebooklm_strict_local=False` + `notebook_provider` not None. Currently masked because `notebooklm_enabled` defaults to `False` and the API router creates `NotebookService` without a provider. Will crash immediately if flags are toggled on.

---

### C2 тАФ Notebook Workflow Handlers Have Wrong Signatures

**File:** `toll/application/handler_registry.py:50-56`  
**Type:** Bug

All other workflow handlers follow the convention `handler(plan: dict, metadata: dict | None)`, but notebook handlers are registered with incompatible signatures:

| Registered method | Expects | Receives from engine | Result |
|---|---|---|---|
| `svc.upload_source` | `(notebook_id, content, file_name, title)` тАФ 4 args | `(plan, metadata)` тАФ 2 dicts | Dict passed as `notebook_id` тЖТ crash |
| `svc.create_notes` | `(notebook_id, source_ids=None)` тАФ 2 args | `(plan, metadata)` | Dict passed as `notebook_id` тЖТ crash |
| `svc.query` | `(notebook_id, question)` тАФ 2 args | `(plan, metadata)` | Dict passed as `notebook_id` тЖТ crash |

The workflow engine dispatches handler `(workflow.plan, workflow.metadata)` at `toll/workflow/engine.py:116`. Every notebook handler will raise `TypeError` at runtime.

---

### C3 тАФ Source Content Never Persisted Locally

**File:** `toll/application/notebook_service.py:380-387`  
**Type:** Data loss / functional gap

`_get_source_content()` queries only `title, file_name, metadata` from `notebook_sources` and concatenates them as `"{title} ({file_name})"`. The actual uploaded `content` text is:
1. Used only for `char_count=len(content)` (line 140)
2. Passed to the external provider (line 143)
3. **Never stored in the database**

Consequences:  
- `_generate_note_content()` passes `"MyDoc (report.txt)"` (300 chars of metadata) to the AI and asks for a summary тЖТ meaningless output  
- `query()` builds a prompt from `"title (filename)"` pairs тЖТ fundamentally broken for locally stored sources  
- No source content is recoverable in `strict_local` mode

## 2. Recommended Findings

### R1 тАФ FTS5 External Content Table Misconfigured for TEXT Primary Key

**File:** `toll/model/migrations/0006_notebooks.sql:59-63`  
**Type:** Design flaw

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS notebook_notes_fts USING fts5(
    title, content, tags,
    content='notebook_notes',
    content_rowid='rowid'
);
```

`notebook_notes` uses `id TEXT PRIMARY KEY`, so its implicit `rowid` is an integer alias of the hidden `rowid` column, NOT the UUID string primary key. The FTS5 `content_rowid='rowid'` expects rows to be synced by integer rowid, but `_index_notes` inserts into `notebook_notes_fts` without specifying a rowid. FTS rows get auto-assigned integer IDs that correspond to nothing in `notebook_notes`.

If the content-sync feature were ever activated (e.g., `INSERT INTO notebook_notes_fts(notebook_notes_fts) VALUES('rebuild')`), the index would be corrupted because the linkage is broken.

---

### R2 тАФ FTS5 Index Populated but Never Queried

**File:** `toll/application/notebook_service.py:389-398`, `toll/model/migrations/0006_notebooks.sql:59-63`  
**Type:** Dead code / wasted writes

`_index_notes()` INSERTs into `notebook_notes_fts` when `notebooklm_memory_index` is enabled, but **no code anywhere queries the FTS table**. The `NotebookService.query()` method constructs a prompt from source titles/filenames and sends it to the AI тАФ it never uses the FTS index. The index is 100% write-only, consuming storage and write time with zero read benefit.

---

### R3 тАФ FTS Entries Never Cleaned Up on Note Deletion

**File:** `toll/application/notebook_service.py:262-272`, `toll/application/notebook_service.py:389-398`  
**Type:** Data bloat

`delete_note()` removes the row from `notebook_notes` but **never deletes** the corresponding FTS rows. Repeated create/delete cycles leave orphaned FTS entries indefinitely. Since the FTS table is never queried (R2), this is silent data bloat with no visible symptoms.

---

### R4 тАФ No Cross-Reference Between Notebook FTS and MemoryGraph

**File:** `toll/application/notebook_service.py:225-226` vs `toll/memory/graph.py`  
**Type:** Architectural gap

The flag `notebooklm_memory_index` FTS-populates `notebook_notes_fts` only тАФ it has zero interaction with the `MemoryGraph` class (`toll/memory/graph.py`), which manages the `memories` table. These are two completely separate systems with:
- Separate tables (no foreign key, no join path)
- No synchronization mechanism
- No overlapping APIs

The flag name `notebooklm_memory_index` is misleading тАФ there is no memory indexing in the MemoryGraph sense.

---

### R5 тАФ `asyncio.run()` in `AI.ask()` Crashes in Async Contexts

**File:** `toll/core/ai.py:32,42`  
**Type:** Threading / environment bug

```python
response = asyncio.run(provider.ask(prompt, system))
```

`asyncio.run()` raises `RuntimeError: asyncio.run() cannot be called from a running event loop` if called from a thread that already has an event loop. Called from `NotebookService.query()` and `NotebookService._generate_note_content()`.

Currently works because FastAPI handlers are sync `def` (run in thread pool). Would break if:
- Any endpoint is converted to `async def`
- A background asyncio task calls these services
- pytest-asyncio test calls them without wrapping

---

### R6 тАФ Missing 404 Responses on Notebook Child Endpoints

**File:** `api/routers/notebooks.py:181-199, 235-253, 299-317`  
**Type:** API correctness

`list_sources`, `list_notes`, and `list_snapshots` return empty lists for non-existent notebook IDs instead of 404. `delete_source`, `delete_note`, `delete_snapshot` always return `{"success": True}` even if the resource never existed.

---

### R7 тАФ XSS Risk from Raw HTML Injection of AI Query Response

**File:** `web/index.html:1421`  
**Type:** Security

```javascript
resultDiv.innerHTML = data.answer;
```

The AI's notebook query response is injected as raw HTML. The system prompt currently generates safe Arabic text, but nothing prevents the AI from producing `<script>`, `<img onerror>`, `<style>`, or `<iframe>` tags that would execute in the browser. Use `textContent` or add a sanitizer.

---

### R8 тАФ NotebookService Dependency Pattern Inconsistent

**File:** `toll/application/notebook_service.py:28-40` vs `toll/application/handler_registry.py:50-56`  
**Type:** Architectural inconsistency

| Service | Dependencies | Provider source |
|---------|-------------|-----------------|
| `CarouselService` | `artifact_service, selector, cm` | Via `ProviderSelector` |
| `ReportService` | `artifact_service, selector, cm` | Via `ProviderSelector` |
| `ResearchService` | `artifact_service, selector, cm, flags` | Via `ProviderSelector` |
| **NotebookService** | **`artifact_service, cm, flags, notebook_provider`** | **Direct injection** |

`NotebookService` bypasses `ProviderRegistry` entirely. The `NotebookLMProvider` is instantiated ad-hoc in `handler_registry.py` instead of being registered with the registry.

---

### R9 тАФ ProviderRegistry Has No Notebook Provider Category

**File:** `toll/core/registry.py:19-29`, `toll/core/provider_selector.py:27`

`ProviderRegistry` manages `_llm` and `_search` provider categories. There is no `_notebook` dict, no `register_notebook()` method, no `get_notebook()` method. `ProviderSelector.select()` only calls `available_llm()`, so it cannot select notebook providers.

## 4. Architectural Health Summary

### Strengths
- **Layered architecture is clean** тАФ `api/` тЖТ `application/` тЖТ `core/` тЖТ `ports/`, no upward or circular dependencies
- **Feature flag coverage is thorough** тАФ 46 flags across all layers, all new capabilities gated behind `notebooklm_* = False` defaults
- **Migration system is solid** тАФ sequential SQL migrations with tracking table, proper `IF NOT EXISTS`, no manual schema changes
- **SQL injection surface is zero** тАФ all queries use parameterized `?` placeholders
- **Test baseline is strong** тАФ 170 passing tests, citation engine and workflow engine tests are thorough
- **Single-user Local-First design is honest** тАФ no pretense of multi-user scalability, `asyncio.run()` and `sync def` handlers are deliberate tradeoffs
- **API surface is comprehensive** тАФ 59 endpoints, 15 dedicated to notebooks, all port operations exposed

### Weaknesses
- **Async contract is entirely broken** тАФ the entire `NotebookPort` declares `async def` methods, but every single caller (service, adapter stub, frontend) treats them as synchronous. Either the port should be sync, or the callers should await.
- **Notebook content persistence is missing** тАФ the most fundamental operation (save uploaded content locally) was never implemented. Note generation and query are effectively broken for local sources.
- **Workflow dispatch for notebooks is broken** тАФ handlers registered with wrong signatures guarantee crashes on dispatch.
- **FTS5 index is write-only dead data** тАФ populated on every note creation, never queried, never cleaned up, misconfigured for the TEXT PK schema.
- **Test gaps are concentrated in the newest code** тАФ notebook service, adapter, API, and research integration all have zero test coverage.
- **Provider layer doesn't cover notebooks** тАФ `NotebookPort` is the only port not registered in `ProviderRegistry`, creating an inconsistent dependency injection pattern.
